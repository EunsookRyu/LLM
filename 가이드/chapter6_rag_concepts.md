# 제6장. RAG — 개념과 전체 파이프라인 이해

## 6.1 왜 RAG가 필요한가

제5장까지의 작업으로 HyperCLOVA X API를 이용한 챗봇이 완성되었다. 프론트엔드와 FastAPI 백엔드가 Docker Compose로 통합되어 실제로 대화할 수 있는 상태다. 이 챗봇은 일상적인 대화나 일반적인 질문에는 충분히 유용하다. 그러나 우리 조직의 교육 프로그램 안내, 운영 매뉴얼, 내부 업무 문서처럼 모델이 학습하지 않은 내용에 대해 물어보면 정확한 답변을 기대하기 어렵다.

이 문제는 LLM의 구조적 한계에서 비롯된다.

**지식의 고정성**이 첫 번째 한계다. LLM은 특정 시점까지의 데이터로 학습을 마친 후 배포된다. 학습 이후에 발생한 사건이나 새로 작성된 문서의 내용은 모델이 알 수 없다. 모델 자체를 다시 학습시키지 않는 한 지식을 갱신할 방법이 없다.

**조직 내부 정보의 부재**가 두 번째 한계다. 외부에 공개되지 않은 내부 문서, 교육 프로그램 상세 안내, 특정 업무 절차 같은 내용은 공개 데이터로 학습하는 LLM이 알 수 없다. 모델에게 이런 정보를 알려주려면 어떤 방식으로든 추가적인 컨텍스트를 제공해야 한다.

**할루시네이션**이 세 번째 한계다. LLM은 자신이 모르는 내용에 대해서도 그럴듯한 답변을 생성하는 경향이 있다. 이를 할루시네이션이라고 한다. 모르는 것을 모른다고 답하는 대신, 그럴듯하게 들리지만 사실과 다른 내용을 만들어내는 것이다. 교육 안내나 업무 절차를 다루는 챗봇에서 이는 심각한 문제가 될 수 있다.

RAG는 이 세 가지 한계를 모두 완화하는 접근 방식이다.

### RAG의 핵심 아이디어

RAG는 Retrieval-Augmented Generation의 약자로, 검색 증강 생성이라고 번역한다. 이름에서 알 수 있듯이 핵심 아이디어는 간단하다. LLM이 답변을 생성하기 전에, 사용자의 질문과 관련된 문서를 먼저 검색하여 그 내용을 LLM에게 제공하는 것이다.

사람이 어려운 질문을 받았을 때 관련 자료를 먼저 찾아보고 답변하는 것과 같은 방식이다. LLM 자체의 지식만으로 답하는 것이 아니라, 신뢰할 수 있는 문서를 참조하여 답변의 정확성을 높인다.

이 방식의 장점은 세 가지다. 첫째, 문서를 교체하거나 추가하는 것만으로 챗봇이 참조하는 지식을 갱신할 수 있다. 모델을 다시 학습시킬 필요가 없다. 둘째, 어떤 문서를 근거로 답변했는지 출처를 함께 제공할 수 있어 신뢰성이 높아진다. 셋째, 조직 내부 문서처럼 외부에 공개되지 않은 정보를 활용할 수 있다.

### RAG로 해결되지 않는 것

RAG가 모든 문제를 해결하는 것은 아니다. RAG는 문서에 있는 내용을 정확히 찾아 전달하는 능력을 높여주지만, LLM의 기본적인 추론 능력을 향상시키지는 않는다. 관련 문서가 존재하지 않는 질문에는 여전히 한계가 있다. 또한 검색 자체가 잘못되면, 즉 질문과 관련 없는 문서가 검색되면 오히려 답변의 품질이 떨어질 수 있다.

## 6.2 RAG 전체 파이프라인

RAG 시스템은 크게 두 개의 독립적인 파이프라인으로 구성된다. 하나는 문서를 미리 처리하여 저장하는 **인덱싱 파이프라인**이고, 다른 하나는 사용자의 질문에 답변하는 **검색 및 생성 파이프라인**이다.

두 파이프라인은 벡터 데이터베이스를 중심으로 연결된다. 인덱싱 파이프라인은 문서를 처리하여 벡터 데이터베이스에 저장하고, 검색 및 생성 파이프라인은 벡터 데이터베이스에서 관련 문서를 찾아 LLM에 전달한다. 제5장에서 Docker Compose에 포함시킨 Qdrant가 이 벡터 데이터베이스 역할을 담당한다.

### 인덱싱 파이프라인

인덱싱 파이프라인은 사용자의 요청과 무관하게 미리 실행된다. 새로운 문서를 시스템에 추가하거나 기존 문서를 업데이트할 때 실행한다.

```
[원본 문서]
    ↓ 문서 로딩
[텍스트 추출]
    ↓ 청킹
[텍스트 조각들]
    ↓ 임베딩 모델
[벡터들]
    ↓ 저장
[벡터 데이터베이스 (Qdrant)]
```

각 단계를 상세히 설명한다.

**문서 로딩**은 PDF, DOCX, TXT 등 다양한 형식의 파일에서 텍스트를 추출하는 단계다. 파일 형식마다 텍스트를 추출하는 방법이 다르므로, 각 형식을 처리하는 로더가 필요하다.

**청킹**은 추출한 텍스트를 적절한 크기의 조각으로 나누는 단계다. LLM이 한 번에 처리할 수 있는 텍스트 양에는 한계가 있으며, 너무 긴 텍스트를 통째로 저장하면 나중에 검색 정확도도 떨어진다. 적절한 크기로 나누는 것이 RAG 성능에 큰 영향을 미친다.

**임베딩**은 텍스트 조각을 숫자 벡터로 변환하는 단계다. 임베딩 모델이 텍스트의 의미를 수백에서 수천 개의 숫자로 표현한다. 의미가 비슷한 텍스트끼리는 벡터 공간에서 가까운 위치에 놓인다. 이 성질을 이용해 나중에 유사한 내용의 문서를 검색할 수 있다.

**저장**은 텍스트 조각, 그에 해당하는 벡터, 그리고 메타데이터를 벡터 데이터베이스에 저장하는 단계다. 메타데이터에는 원본 파일 이름, 페이지 번호, 작성일 같은 부가 정보가 포함된다. 나중에 검색 결과를 필터링하거나 출처를 표시할 때 사용한다.

### 검색 및 생성 파이프라인

검색 및 생성 파이프라인은 사용자가 질문할 때마다 실시간으로 실행된다.

```
[사용자 질문 — 프론트엔드에서 전송]
    ↓ 임베딩 모델
[질문 벡터]
    ↓ Qdrant 유사도 검색
[관련 문서 조각들]
    ↓ 프롬프트 구성 (FastAPI 백엔드)
[컨텍스트가 포함된 프롬프트]
    ↓ LLM (HyperCLOVA X 또는 로컬 LLM)
[최종 답변 → 프론트엔드로 스트리밍]
```

**질문 임베딩**은 사용자의 질문을 인덱싱 때 사용한 것과 동일한 임베딩 모델로 벡터로 변환하는 단계다. 동일한 모델을 사용해야 벡터 공간에서의 비교가 의미 있다.

**유사도 검색**은 질문 벡터와 데이터베이스에 저장된 문서 벡터들을 비교하여 가장 유사한 문서 조각들을 찾아내는 단계다. 이때 사용하는 수학적 기법이 코사인 유사도다.

**프롬프트 구성**은 검색된 문서 조각들을 LLM에 전달할 프롬프트에 포함시키는 단계다. LLM에게 "다음 문서를 참고하여 질문에 답하세요"라는 형태로 컨텍스트를 제공한다. 이 과정은 FastAPI 백엔드의 서비스 레이어에서 처리되며, 프론트엔드는 이 과정을 알 필요가 없다.

**생성**은 컨텍스트가 포함된 프롬프트를 LLM에 전달하여 최종 답변을 얻는 단계다. LLM은 자신의 일반 지식과 제공된 컨텍스트를 결합하여 답변을 생성한다.

## 6.3 임베딩 이해하기

임베딩은 RAG의 핵심 기술이다. 임베딩을 이해하지 못하면 RAG 시스템의 동작 원리를 파악하기 어렵다.

### 텍스트를 숫자로 표현한다는 것

컴퓨터는 텍스트의 의미를 직접 이해하지 못한다. 컴퓨터가 처리할 수 있는 것은 숫자다. 임베딩은 텍스트를 숫자 벡터로 변환하는 과정이다. 단순한 인코딩이 아니라, 텍스트의 의미를 보존하는 방식으로 변환한다.

예를 들어 "강아지"와 "반려견"은 서로 다른 단어지만 의미가 유사하다. 임베딩 모델은 이 두 단어를 벡터 공간에서 서로 가까운 위치에 배치한다. 반면 "강아지"와 "자동차"는 의미가 다르므로 벡터 공간에서 멀리 떨어진 위치에 놓인다.

임베딩 벡터는 수백에서 수천 개의 숫자로 구성된다. 예를 들어 1024차원의 벡터라면 1024개의 숫자로 하나의 텍스트를 표현한다. 각 차원이 텍스트의 어떤 의미적 특성을 나타내는지는 사람이 직접 해석하기 어렵지만, 모델은 이 고차원 공간에서 텍스트 간의 의미적 유사성을 수치로 계산할 수 있다.

### 코사인 유사도

두 벡터의 유사도를 계산하는 방법 중 RAG에서 가장 일반적으로 사용되는 것이 코사인 유사도다. 코사인 유사도는 두 벡터가 이루는 각도의 코사인 값으로, -1에서 1 사이의 값을 가진다. 값이 1에 가까울수록 두 벡터가 같은 방향을 가리키고 있다는 의미이며, 의미가 유사하다는 뜻이다. 값이 0에 가까울수록 두 벡터는 서로 무관한 내용을 담고 있다.

검색 과정에서는 사용자 질문의 벡터와 데이터베이스에 저장된 모든 문서 조각의 벡터를 비교하여, 코사인 유사도가 높은 순서대로 문서 조각을 정렬하고 상위 K개를 반환한다. 이를 Top-K 검색이라고 한다.

### 한국어 임베딩 모델

임베딩 모델의 품질은 RAG 성능에 직접적인 영향을 준다. 한국어 텍스트를 처리하는 경우 한국어를 잘 이해하는 임베딩 모델을 선택하는 것이 중요하다.

이 가이드에서는 두 가지 임베딩 옵션을 사용한다.

**HyperCLOVA X 임베딩 API**는 1단계와 2단계에서 사용한다. CLOVA Studio에서 제공하는 임베딩 API로, 외부 네트워크를 통해 호출한다. 추가 설치 없이 바로 사용할 수 있다는 장점이 있다.

**BGE-M3**는 3단계에서 사용한다. BAAI에서 개발한 다국어 임베딩 모델로, 한국어 성능이 우수하다. 로컬에서 직접 실행할 수 있어 외부 API 의존도를 완전히 없앨 수 있다. 1024차원의 벡터를 생성한다.

중요한 점은 인덱싱과 검색에 반드시 **동일한 임베딩 모델**을 사용해야 한다는 것이다. 인덱싱 시 HyperCLOVA X 임베딩으로 저장한 벡터를 BGE-M3로 검색하면 의미 있는 유사도 비교가 불가능하다. 3단계에서 임베딩 모델을 교체할 때는 기존에 저장된 모든 문서를 새 모델로 재인덱싱해야 한다.

## 6.4 벡터 데이터베이스 이해하기

벡터 데이터베이스는 고차원 벡터를 효율적으로 저장하고 빠르게 유사도 검색할 수 있도록 설계된 특수한 데이터베이스다. 일반적인 관계형 데이터베이스는 정확한 값을 찾는 데 최적화되어 있지만, 벡터 데이터베이스는 "이 벡터와 가장 유사한 것들을 찾아라"라는 근사 최근접 이웃(ANN) 검색에 특화되어 있다.

수만 개의 문서 조각이 저장된 데이터베이스에서 가장 유사한 벡터를 찾으려면 이론적으로는 모든 벡터와 일일이 비교해야 한다. 벡터 데이터베이스는 HNSW나 IVF 같은 인덱싱 알고리즘을 사용하여 전수 비교 없이도 빠르게 가장 유사한 벡터를 찾는다.

### Qdrant

이 가이드에서는 **Qdrant**를 벡터 데이터베이스로 사용한다. 제5장의 Docker Compose에 이미 Qdrant 서비스를 포함시켜 두었으므로, 별도 설치 없이 바로 사용할 수 있는 상태다.

Qdrant를 선택한 이유는 다음과 같다.

- **Rust로 작성**되어 성능이 뛰어나다. 수백만 개 이상의 벡터도 안정적으로 처리한다.
- **REST API와 gRPC**를 통해 접근하며, Python 클라이언트 라이브러리도 제공한다.
- **메타데이터 필터링**을 지원하여 "특정 문서에서만 검색" 같은 조건을 쉽게 적용할 수 있다.
- **Docker 이미지**로 배포가 간편하고, 웹 대시보드(`http://localhost:6333/dashboard`)를 내장하고 있다.
- 폐쇄망 환경에서도 Docker 이미지만 있으면 실행할 수 있어 이 프로젝트의 요구사항에 부합한다.

> **참고 — ChromaDB**: 벡터 데이터베이스를 처음 접한다면 ChromaDB로 실습해 볼 수도 있다. `pip install chromadb`만으로 설치되고 별도 서버 없이 로컬 파일로 동작하여 학습 목적으로 적합하다. 다만 이 가이드에서는 처음부터 운영 환경을 염두에 두고 Qdrant를 사용한다.

### 메타데이터의 역할

벡터 데이터베이스에 문서 조각을 저장할 때는 벡터와 함께 원본 텍스트와 메타데이터도 함께 저장한다. 메타데이터는 검색 결과를 필터링하거나 출처를 표시하는 데 사용된다.

예를 들어 다음과 같은 메타데이터를 저장할 수 있다.

```json
{
  "source": "사내_규정집_2025.pdf",
  "page": 15,
  "chapter": "제3장 휴가 및 복무",
  "created_at": "2025-01-15",
  "category": "규정"
}
```

검색 시 `source`가 특정 파일인 경우만 검색하거나, `category`가 "교육자료"인 문서만 검색하는 필터를 적용할 수 있다. Qdrant는 이러한 메타데이터 필터링을 벡터 검색과 동시에 수행하여 정밀한 검색이 가능하다.

## 6.5 청킹 전략

청킹은 RAG 성능에 가장 큰 영향을 미치는 요소 중 하나다. 문서를 어떻게 나누느냐에 따라 검색 정확도가 크게 달라진다.

### 청크 크기의 트레이드오프

청크 크기는 검색 정확도와 컨텍스트 풍부함 사이의 균형을 결정한다.

청크가 너무 작으면, 하나의 청크만으로는 질문에 답하기 충분한 정보를 담기 어렵다. 문맥이 잘리면 임베딩의 의미 표현도 부정확해진다.

청크가 너무 크면, 하나의 청크 안에 여러 주제의 내용이 섞이게 된다. 임베딩이 특정 주제를 제대로 표현하지 못하여 검색 정확도가 떨어진다. 또한 LLM에 전달하는 컨텍스트 크기가 커져 비용과 처리 시간이 증가한다.

일반적으로 권장하는 청크 크기는 512에서 1024 토큰 사이다. 하지만 문서의 성격에 따라 달라진다. 교육 프로그램 안내처럼 항목이 독립적인 의미를 가지는 문서는 항목 단위로 나누는 것이 효과적이다. 기술 문서나 긴 설명문은 더 큰 청크가 적합할 수 있다.

### 청크 오버랩

청크를 나눌 때 인접한 청크 사이에 일정 부분을 겹치게 하는 것을 오버랩이라고 한다. 오버랩을 두는 이유는 중요한 정보가 청크 경계에서 잘리는 것을 방지하기 위함이다.

예를 들어 청크 크기가 500 토큰이고 오버랩이 100 토큰이라면, 두 번째 청크는 첫 번째 청크의 마지막 100 토큰을 포함하여 시작한다. 청크 경계 근처의 내용이 어느 한쪽 청크에만 들어가지 않고 양쪽 청크에 모두 포함된다.

일반적으로 청크 크기의 10에서 20% 수준의 오버랩을 사용한다.

### 주요 청킹 방식

**고정 크기 청킹**은 문자 수나 토큰 수를 기준으로 일정하게 나누는 방식이다. 구현이 간단하고 예측 가능하지만, 문장이나 문단의 경계를 무시하고 자를 수 있다는 단점이 있다.

**재귀적 청킹**은 단락, 줄바꿈, 문장 부호 순서로 경계를 찾아 나누는 방식이다. 가능하면 자연스러운 경계에서 나누되, 해당 경계에서 나눈 결과가 너무 크면 다음 단계의 구분자로 넘어간다. 대부분의 경우에 가장 균형 잡힌 결과를 제공하여 가장 널리 사용된다.

**시맨틱 청킹**은 임베딩을 이용하여 의미가 전환되는 지점을 찾아 나누는 방식이다. 의미적으로 응집된 청크를 만들 수 있지만, 처리 비용이 높고 구현이 복잡하다. 품질이 중요한 경우에 고려한다.

이 가이드에서는 재귀적 청킹 방식을 기본으로 사용한다.

## 6.6 RAG 프롬프트 설계

검색된 문서를 LLM에 전달하는 방식, 즉 RAG 프롬프트의 구조도 중요하다. 잘 설계된 프롬프트는 LLM이 문서의 내용을 정확히 활용하도록 유도한다.

RAG 프롬프트의 기본 구조는 다음과 같다.

```
[시스템 프롬프트]
당신은 조직의 내부 문서를 기반으로 질문에 답변하는 AI 어시스턴트입니다.
아래 제공된 참고 문서의 내용을 우선적으로 활용하여 질문에 정확하게 답변하세요.
참고 문서에 없는 내용은 모른다고 솔직하게 답변하세요.
답변 마지막에는 참고한 문서의 출처를 표시하세요.

[검색된 참고 문서]
--- 문서 1 ---
출처: 사내_규정집_2025.pdf (15페이지)
내용: ...검색된 텍스트 조각 1...

--- 문서 2 ---
출처: 업무_매뉴얼_v2.pdf (3페이지)
내용: ...검색된 텍스트 조각 2...

[사용자 질문]
연차 휴가 신청 절차는 어떻게 되나요?
```

이 프롬프트 구성은 FastAPI 백엔드의 서비스 레이어에서 처리된다. 프론트엔드는 사용자의 메시지만 전송하고, 백엔드가 내부적으로 Qdrant 검색 → 프롬프트 구성 → LLM 호출을 수행한 뒤 OpenAI 호환 형식으로 응답을 반환한다. 프론트엔드 입장에서는 RAG가 적용되었는지 여부를 알 필요가 없다.

이 구조에서 중요한 점이 몇 가지 있다.

"참고 문서에 없는 내용은 모른다고 답변하세요"라는 지시는 할루시네이션을 억제하는 데 중요하다. 이 지시가 없으면 LLM이 문서에 없는 내용을 자신의 지식으로 채워 넣는 경우가 생긴다.

출처를 표시하도록 요청하면 사용자가 답변의 근거를 확인할 수 있어 신뢰성이 높아진다.

참고 문서를 구분선(`---`)으로 명확히 구분하면 LLM이 각 문서의 경계를 쉽게 인식한다.

## 6.7 RAG 성능을 결정하는 요소들

RAG 시스템을 구축하면서 성능에 영향을 미치는 주요 요소들을 미리 알아 두면 문제가 발생했을 때 원인을 파악하기 쉽다.

**임베딩 모델의 품질**이 가장 근본적인 요소다. 임베딩 모델이 텍스트의 의미를 잘 포착하지 못하면, 아무리 나머지 구성 요소를 잘 만들어도 검색 정확도를 높이는 데 한계가 있다.

**청크의 품질**이 두 번째로 중요한 요소다. 청크 크기와 방식이 적절하지 않으면 중요한 정보가 분리되거나 너무 많은 노이즈가 섞인다.

**Top-K 값**은 검색 시 몇 개의 문서 조각을 반환할지를 결정한다. 값이 너무 작으면 관련 정보가 검색되지 않을 수 있고, 너무 크면 관련 없는 내용이 LLM에 전달되어 오히려 품질이 떨어질 수 있다. 일반적으로 3에서 5 사이의 값을 사용한다.

**문서의 품질**도 중요하다. 스캔 품질이 낮은 PDF, 표나 이미지가 많은 문서처럼 텍스트 추출이 어려운 문서는 RAG 성능을 저하시킨다. 가능하면 텍스트 기반의 문서를 사용하고, 전처리 단계에서 불필요한 노이즈를 제거하는 것이 좋다.

## 6.8 2단계 구현 계획

RAG의 개념을 충분히 이해했으니, 이제 실제 구현 계획을 정리한다. 제5장에서 Qdrant가 이미 Docker Compose에 포함되어 실행 중이므로, 인프라 준비 없이 바로 구현에 돌입할 수 있다.

제7장에서는 Qdrant의 기본 사용법을 익힌다. Python 클라이언트를 사용하여 컬렉션을 생성하고, 벡터를 저장하고 검색하는 기본 작업을 실습한다. Qdrant 대시보드(`http://localhost:6333/dashboard`)를 활용하여 저장된 데이터를 시각적으로 확인하는 방법도 다룬다.

제8장에서는 문서를 불러와 청킹하고 임베딩 벡터를 생성하여 Qdrant에 저장하는 인덱싱 파이프라인을 구축한다.

제9장에서는 사용자 질문을 받아 관련 문서를 검색하고 LLM에 전달하는 검색 및 생성 파이프라인을 구현하여 기존 챗봇에 통합한다. 이 과정에서 백엔드의 `/v1/chat/completions` 엔드포인트에 RAG 로직을 추가하되, 프론트엔드(예: NextChat, Streamlit 등)와의 인터페이스는 변경하지 않는다.

지금 당장 이 모든 내용을 완전히 이해하지 못해도 괜찮다. 실제 코드를 작성하고 시스템이 동작하는 것을 확인하면서 개념이 명확해질 것이다. 이 장에서 설명한 내용은 이후 각 장을 읽을 때 참고 자료로 활용한다.
